# SRT 実行時ルール正典（v6）— Step 5 はこのファイルだけ読めばよい

`/srt` / `/srt-fast` の改行工程（LLM が lines.txt を書く工程）で参照する**唯一の実行時ルール**。
過去全セッションの修正指示をここに蒸留済み。

- 原典（履歴・根拠・完全な辞書表）: `memory/feedback_srt_grouping_rules.md` / `memory/telop_channel_patterns.md`
- 数値が原典間で食い違う場合は**このファイルが正典**（矛盾は統一済み）
- 新しい失敗パターンが出たら: memory 側に経緯を追記 → このファイルに**ルールだけ**反映

## 0. 責務範囲（この5原則が全ルールに優先）

1. **区切り位置の判断は必ず LLM が全文を読んで行う**。fugashi / MeCab / Ginza / 品詞ベースのルールは候補抽出の補助までで、`should_emit_between()` のような機械 emit 判定を書くのは禁止（v2 で「思って/いるので」分断を量産した事故の教訓）
2. **喋りの意味単位で区切る**。編集者の物理的再分割（1行に収まらないから切る）は模倣しない。文字数ハード制約を入れない
3. **喋った全行を出力する**。不採用判定・行の省略をしない（挨拶・列挙・CTAも残す。省くと後工程の時刻が狂う）
4. **読点「、」句点「。」は使わない**。列挙は中黒「・」、区切りは半角スペース
5. **可読性のための機械的なスペース挿入・一律削除をしない**。フィラーも表記も最終判断は文脈を読んで行う

## 1. 絶対禁止（1件でも該当したら修正必須）

| # | 禁止 | 例 |
|---|---|---|
| 1 | **文頭フラグメント**: 助詞単独（か・も・を・に・が・は・の・と・や・ね・よ・ぞ）、単独「で」、活用語尾（ます・まし・ません・でした・きます等）、否定（ない・なく・なっ）、複合動詞後半（きており・きました）、連体接続（という・として・について・によって）、抽象名詞+助詞（ため・とき・はず・こと・もの）、カタカナ破片（ベル・ース・ション・ング等） | ✗「いるので」✗「か2026年」→前の行に結合 |
| 2 | **複合動詞句の分断**: 〜ている/ていく/てくる/てみる/ておく/てしまう/てほしい/てくれる/てあげる/てもらう/てやる/てもいい/てはいけない/てこそ ＋派生形（〜ている所/ていこう/ておりました等）は必ず1行 | ✗「思って/いるので」✓「思っているので」 |
| 3 | **動詞活用の中断**: 語幹と活用語尾を切らない | ✗「でき/ます」「し/ている」 |
| 4 | **「た＋連体修飾名詞」の分断**: た+こと/もの/とき/はず/ため は一体 | ✓「AIの音声だったことに気づきましたか」 |
| 5 | **「ということで/は/です」の分断**: 複合終止表現として一体 | ✓「なくなったんじゃないかということで」 |
| 6 | **カタカナ語・英数字・固有名詞の分断**: チャンネル名・製品名（例:「MiniMax Audio」「LINE登録」）等は一体 | ✗「レ/ベル」 |
| 7 | **終助詞「か」の切り離し**: 質問は「か」まで含めて終わらせる | ✓「気づきましたか」 |
| 8 | **単独で意味が通じない行**: 各行を単独で読んで視聴者が内容を把握できること | ✗「ことに」 |

## 2. 積極的に切る（ソフト優先・「繋げられるから繋げる」は誤り）

人間編集はこちらより**33件/動画ほど細かく**分ける。迷ったら切る。

- **文末（です/ます）＋新トピック開始**で切る
- **「〜おり/ており/まして」で終止**して切る。後続の主語は**次行の頭**に置く（✓「レベルになってきており」+「人間が話す必要は〜」）
- **「ですけども/なんですけど/ですが」の後**で切る。フィラー的接続でも独立行にする（✓「今回なんですけども」+「より皆さんが〜」）
- **程度副詞（徹底的に/本当に/しっかりと/きちんと）は次行の先頭**（後続動詞に係る）（✓「〜のを」+「徹底的に検証していきたいと思います」）
- **「〜ではなく」＋新主語**で切る
- **話題転換では必ず切る**: 説明→CTA（LINE登録・概要欄・QR誘導）／概念→具体例（例えば・こちら・これは）／抽象→数値（なんと○万文字）／肯定→疑問（〜か？）
- 4〜8字の短い行も**意味の区切りならOK**（✓「人間が話す必要が」8字）

## 3. 削除・保持（LLM の文脈判断。機械的な一律処理は禁止）

**削除候補**（積極的に削る）:
- メタコメント（「これを紹介したいだけがために〜」「〜を見せたかったから」）
- 時間稼ぎ（「いいかなという感じはしますね」「近づいてるなというのを感じました」）
- ナレーション埋め草（「じゃあこのまま生成していきます」）
- 主観の冗長形（「〜というふうに僕は思ってるんですけども」→「〜と思います」で十分）
- 相槌の繰り返し（「そうそうそう」→1回に圧縮 or 削除）

**保持**（削りすぎ注意・過去に削りすぎで指摘された）:
- 「**こう**」「**ちょっと**」「**確かに**」— 副詞的用法・強調は意味を持つ。全削除禁止
- 「〜**ですね**/ますね」— 共感マーカー。基本残す
- 「〜ですよ/ですよね」— 強調マーカー。残す方向、冗長なら削除可
- フィラー「ちまちま」等も発話通り残す（編集者は強調テロップに移すだけで消していない）

## 4. 文字数・時間の目標値（正典）

| 指標 | 目標 |
|---|---|
| 平均文字数 | **14字前後**（最頻 11〜15字） |
| 25字超 | **1%未満**（超えたら分割を検討。ただし変な位置で切るくらいなら許容） |
| 4字未満 | 最小限（意味の区切りなら可） |
| 表示時間 | 0.8〜3.5秒（スクリプトが最小800ms保証） |
| 文字/秒 | 5〜10（平均7前後） |
| 行数 | 1行固定。改行は絶対に入れない |

## 5. 半角スペース（使用率は全体の8%程度が目安）

入れる場所は3類型のみ:
1. **話者のポーズ・息継ぎ**（目安: 一拍 ≒ 300ms 以上の間）: 「実は今 喋っていたのは」「結論 これが」
2. **日本語↔英数字の境界**で読みやすくなるとき: 「今回は本当に AI音声は」「Sad 悲しいにしておきましょう」
3. **英語プロダクト名・UI文字列の内部正規表記**: `MiniMax Audio` `Fish Audio` `Sound Tag` `Generation completed successfully`

助詞の前後には入れない。ひらがな連続の可読性調整のための機械挿入はしない。

## 6. 固有名詞・表記

**チャンネル固有の確定表記は `config/channel_profile.md` と `config/corrections.local.json` に持つ**
（`/srt-fast` の Step 0 セットアップ対話で生成される。空のテンプレは
`config/channel_profile.example.md` / `config/corrections.example.json`）。改行エージェントは
Step 1 でこのファイルがあれば読み、記載された表記ルールを適用する。無ければこのセクションの
運用ルールだけに従う。

**運用ルール**:
- 英語プロダクト名は**正規表記＋内部半角スペース**（`MiniMaxAudio` にしない）。UIラベル・ボタン名は**英語のまま**（Generate / Lyrics 等）
- スクリプトの CORRECTIONS 辞書（`config/corrections.local.json` から読み込み）が既知の誤認識を自動修正する。lines.txt では**残った誤認識を文脈で修正してよい**（v6 は全体アライメントなので lines 側の修正で時刻はズレない）
- **辞書に追加するのは「チャンネル内で再登場する固有名詞」だけ**。一回限りの文脈誤認識は辞書に入れず、その場で文脈修正のみ
- 同音異義語は文脈で判断して修正

**スクリプトが自動適用する正規化**（lines.txt で重ねてやらなくてよい）: っていう→という、4桁数字カンマ（1,200）、漢数字+量詞→算用数字（一番→1番）

## 7. 冒頭のキャリブレーション例（毎回この粒度感で）

実データで精度を上げたい場合は、生成したSRTとPremiereで手直しした完成版を比較し、
差分から得た自チャンネルの正解例をこのセクションに追記していくとよい
（`memory/feedback_srt_grouping_rules.md` の学習ワークフロー参照）。以下は判断基準を示す一般例:

| 正解の行 | 字数 | 判断 |
|---|---|---|
| 皆さんこんにちは今日の配信者です | 16 | 「です」文末＋次が新トピック |
| 今回はAI音声が進化しすぎて | 14 | 連用中止＋次が新主語 |
| 人間が話す必要が | 8 | 「が」終わりでも意味的に完結 |
| なくなったんじゃないかということで | 17 | 複合終止「ということで」一体 |
| 本当にそうなのかを検証していきたいと思います | 22 | 「ます」文末 |
| 実は今 喋っていたのは私の声ではなく | 18 | 「ではなく」連用中止＋ポーズのスペース |
| AIの音声だったことに気づきましたか | 18 | 「か」終助詞＋「た+こと」一体 |
| 2026年のAI音声は | 11 | 「は」で主題提示 |

## 8. 出力形式（lines.txt）

- 1行 = 1テロップ。空行 = 段落区切り（SRT には反映されない）
- 見出し・番号・コメントを書かない。テロップ本文だけを書く

## 9. 学習ワークフロー（継続改善サイクル）

1. `/srt` で生成
2. Premiere Pro でテロップを手動微調整
3. 修正済み SRT + WAV + XML を `@修正済みSRT @WAV @XML` で渡す
4. 差分分析 → `memory/feedback_srt_grouping_rules.md` + `memory/telop_channel_patterns.md` に蓄積
5. 次回の生成に反映

## 10. トラブルシューティング

| エラー | 対処 |
|---|---|
| `faster-whisper が見つかりません` | `pip3 install --user faster-whisper` |
| Whisperタイムアウト | Bashタイムアウトを 600000ms に設定 |
| Whisperハリュシネーションループ | `condition_on_previous_text=False` を確認 |
| SRT が Premiere で文字化け | UTF-8 BOM + CRLF か品質チェックで確認 |
| XMLパースエラー | Premiere から「Final Cut Pro XML」で再書き出し（prproj は不可） |
| `segments.json が見つからない` | Step 3（Whisper 転写）を再実行 |

対応環境: macOS / Linux（Windows は未検証）、Python 3.9 以降、Claude Code CLI、faster-whisper（または mlx-whisper） + ffmpeg。

## 11. 推奨WAV書き出し設定

**重要**: WAV は Premiere Pro タイムラインの **OP 以降** の音声を書き出すこと。

| 項目 | 推奨値 | 備考 |
|---|---|---|
| 書き出し範囲 | OP終了〜本編終了 | WAV の 0 秒 = タイムラインの OP 終了位置 |
| サンプルレート | 16,000 Hz | Whisper 内部処理と同一 |
| チャンネル | モノラル | ステレオ不要 |
| サンプルサイズ | 16-bit | 24-bit は不要 |
